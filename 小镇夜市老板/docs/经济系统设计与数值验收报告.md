# 《小镇夜市老板》经济系统设计与数值验收报告

日期：2026-06-20

## 结论

本轮已把金币从“单局数值增长”改成“经营净利润闭环”：关卡营业赚金币，金币投入菜品、设备、店铺和轻量装扮，反向提升售价、产能、耐心、成本控制、客诉减免和口碑，再挑战更高客流与更高目标。项目仍保持仅 IAA 广告变现，无支付、无现金回流、无随机奖励售卖入口。

自动化验收结果：

- `npm run validate:configs`：通过。
- `npm run test:mvp`：前 1-10 关通过。
- `npm run check:economy`：前 1-30 关推荐升级路线通过。
- `npm run typecheck`：通过。

## 发现的问题

1. 金币来源只有售价、快速服务、连击和固定服务金币，缺少食材成本、客诉赔付和成本控制，玩家赚到钱以后缺少经营判断。
2. 菜品升级只有线性售价成长，玩家感知不到“产品变高级了”。
3. 设备升级只有线性减时和过早工位扩容，产能突破节点不清晰。
4. 缺少店铺升级，金币无法解释为“摊子变亮、顾客更愿意等、食材损耗更低”。
5. 结算广告原本是高倍率追加，容易压过经营升级。
6. 存档缺少店铺、装扮、每日广告奖励和云同步字段。
7. 前 11-30 关金币目标按旧毛收入口径设置，引入净利润后会形成不合理卡关。
8. 自动化只覆盖前 10 关，缺少前 30 关经济回归。

## 经济公式

关卡单客净金币：

```text
净金币 = 菜品收入 + 小费 + 固定服务金币 - 食材成本 - 客诉/错误惩罚
菜品收入 = 菜品售价 * 顾客倍率 * 关卡收益倍率
小费 = 菜品售价 * (快速服务率 + 连击率 + 小费加成) * 顾客倍率 * 关卡收益倍率
食材成本 = 基础售价 * 食材成本率 * 等级成本压力 * (1 - 成本减免)
```

升级属性解释：

| 属性 | 作用 |
|---|---|
| `priceBonus` | 提高全局或指定来源售价 |
| `speedBonus` | 缩短制作时间 |
| `costReduce` | 降低食材成本 |
| `patienceBonus` | 提高顾客最大耐心 |
| `complaintReduce` | 降低超时离开、错误上菜的惩罚 |
| `rating` | 作为口碑，轻微增加服务金币，并预留高端顾客门槛 |

## 广告经济

| 广告入口 | 当前规则 | 经济约束 |
|---|---|---|
| 结算追加金币 | 成功后可看一次广告追加金币 | `min(本局收益*50%, 1星目标*60%)`，向下取整到 10 |
| 失败续时 | 失败且可恢复时看广告续时 15 秒 | 每关一次，写入 `failExtendWatchedByLevel` |
| 每日奖励 | 首页每日广告奖励 mock | 每自然日一次，固定金币，不随机 |

广告不会改变菜品、设备、店铺成长曲线，只作为加速器。

## 云存档方案

客户端只读写本地存档和后端 API，不连接数据库。

推荐 API：

| 方法 | 路径 | 说明 |
|---|---|---|
| `GET` | `/api/v1/save` | 拉取当前玩家云存档 |
| `PUT` | `/api/v1/save` | 上传完整存档，带 `revision` 乐观锁 |
| `POST` | `/api/v1/save/merge` | 多端冲突时提交本地与云端版本，由后端合并 |
| `POST` | `/api/v1/ad-events` | 记录激励广告完成事件，后端可幂等校验 |

MySQL 表建议：

```sql
CREATE TABLE player_save (
  player_id VARCHAR(64) PRIMARY KEY,
  schema_version INT NOT NULL,
  revision BIGINT NOT NULL,
  save_json JSON NOT NULL,
  created_at DATETIME NOT NULL,
  updated_at DATETIME NOT NULL
);

CREATE TABLE ad_event (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  player_id VARCHAR(64) NOT NULL,
  event_key VARCHAR(128) NOT NULL,
  ad_slot VARCHAR(64) NOT NULL,
  level_id INT NULL,
  reward_coins INT NOT NULL DEFAULT 0,
  created_at DATETIME NOT NULL,
  UNIQUE KEY uniq_player_event (player_id, event_key)
);
```

冲突策略：

- `revision` 相同：直接覆盖云端，`revision + 1`。
- `revision` 落后：后端按最高关卡、最高星级、最高升级等级、金币取“云端金币 + 本地新增净收益 - 本地新增花费”的事件流合并；早期可先返回 409，让客户端拉取后重试。
- 广告奖励使用 `event_key` 幂等，例如 `settlement:level:10`、`fail_extend:level:10`、`daily:2026-06-20`。
